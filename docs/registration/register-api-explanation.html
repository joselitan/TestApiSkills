Register-funktionen förklarad

Steg-för-steg-genomgång av @app.route("/api/register", methods=["POST"]) i src/app.py från rad 562.

Funktionssignatur & inställning

Ruttdefinition

@app.route("/api/register", methods=["POST"])

Funktion: def register():

Plats: src/app.py, börjar vid rad 562

Inledande kontroller
  • Kontrollerar om ENABLE_SELF_REGISTRATION är aktiverat (returnerar 403 om det är inaktiverat)
  • Parsar JSON-data från begäran
  • Validerar JSON-struktur och innehåll
  • Loggar varningar för ogiltig JSON
Säkerhet först: Funktionen validerar omedelbart indata och kontrollerar konfiguration innan någon användardata bearbetas.

Inmatningsvalidering & bearbetning

Extrahering av obligatoriska fält
  • email = normalize_email(data.get("email")) - normaliserar till gemener
  • password = data.get("password")
  • confirm_password = data.get("confirm_password")
  • role = data.get("role", app.config["DEFAULT_ROLE"]) - standard är "member"
  • username = data.get("username") - valfritt, genereras automatiskt om det saknas
Logik för att generera användarnamn
  • Om inget användarnamn anges, hämtas det från e-post: email.split("@")[0]
  • Säkrar unikhet genom att lägga till siffror vid behov
  • Begränsar längden till 100 tecken
Valideringsregler
  • Alla obligatoriska fält måste vara närvarande
  • E-post måste ha giltigt format
  • Lösenord måste matcha och vara minst 8 tecken
  • Roll måste finnas i listan av tillåtna roller
  • Admin-rollen är uttryckligen förbjuden
  • Användarnamnet får vara högst 100 tecken
Dubbelkontroll: Kontrollerar om e-postadressen redan finns i databasen innan processen fortsätter.

Databasoperationer

Skapa användare
  • Genererar lösenordshash med werkzeug.security.generate_password_hash
  • Skapar verifieringstoken för e-post med itsdangerous.URLSafeTimedSerializer
  • Sätter utgångstid för verifiering (24 timmar som standard)
  • Sätter in användarpost med is_active = 0
Databasschema

Infogar i users-tabellen följande fält:

  • username
  • email
  • password (hashed)
  • role
  • is_active (set to 0)
  • verification_token
  • verification_expires_at
  • created_at, updated_at
Användare skapas som inaktiva för att förhindra omedelbar inloggning innan e-postverifiering är slutförd.

E-postsändning och svar

E-postförsök
  • Anropar send_verification_email(email, verification_token)
  • Denna funktion bygger verifierings-URL och anropar send_email()
  • send_email() kontrollerar SMTP-konfiguration
  • Om SMTP saknas men EMAIL_DEBUG=true, loggas e-postens innehåll istället
Svarshantering
  • Om e-post skickas lyckosamt: returnerar "Verification email sent. Check your inbox."
  • Om e-post misslyckas: returnerar "Account created, but verification email could not be sent. Check email SMTP configuration."
  • Returnerar alltid HTTP 201 (Created) eftersom användarkontot skapats framgångsrikt
Felförlåtande beteende: Kontoskapandet lyckas även om e-postleveransen misslyckas, vilket möjliggör manuell verifiering eller administrativ aktivering.

Felhantering och säkerhet

Exception Handling
  • Catches JSON decode errors
  • Logs warnings for invalid requests
  • Validates all inputs to prevent injection
  • Sanitizes HTML in usernames and emails
Säkerhetsåtgärder
  • Indatarening med html.escape
  • Lösenordshashning med PBKDF2
  • Tokenbaserad verifiering med utgångstid
  • Rollbaserade begränsningar
  • Förebyggande av dubbla e-postadresser
Loggning: Alla operationer loggas för felsökning och säkerhetsövervakning.

Flödesöversikt

Slutgiltig körväg
  1. Kontrollera att självregistrering är aktiverad
  2. Parsa och validera JSON-payload
  3. Extrahera och validera obligatoriska fält
  4. Generera/automatiskt generera användarnamn
  5. Kolla efter dubblett av e-post
  6. Hasha lösenord och skapa verifieringstoken
  7. Infoga inaktiv användare i databasen
  8. Försök skicka verifieringsmail
  9. Returnera framgångssvar med e-poststatus
Funktionen prioriterar kontoskapande samtidigt som den hanterar e-postleveransproblem på ett felförlåtande sätt.
