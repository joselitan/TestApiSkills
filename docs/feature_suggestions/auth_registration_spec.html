🧭 Specifikation: Användarregistrering & Inloggning

QA Platform - Self-service account creation, email verification och JWT login

Version: v2 proposal | Datum: 2026-04-04

📌 Mål

Ge nya användare möjlighet att själva registrera sig, minska administrativ börda och skapa en säkrare onboarding med verifiering via e-post. Funktionen ska återanvända befintlig JWT-login och lösenordshantering i src/app.py.

🔧 Återanvändning av befintlig logik

Befintliga funktioner i src/app.py ger redan:
  • JWT-tokenautentisering
  • Lösenordskontroll med werkzeug.security.check_password_hash
  • Databaskoppling via get_db()
  • Validering och XSS-skydd med validate_json_payload och sanitize_html
  • Globala säkerhetsheaders
Den nya registreringsflödet kompletterar dessa genom att lägga till:
  • Skapa användare
  • Verifierings-token
  • E-postaktivering
  • Rate limiting och CSRF-stöd

🧱 Databasmodell

CREATE TABLE IF NOT EXISTS users ( id INTEGER PRIMARY KEY AUTOINCREMENT, email TEXT UNIQUE NOT NULL, username TEXT UNIQUE, password TEXT NOT NULL, role TEXT NOT NULL DEFAULT 'member', is_active BOOLEAN NOT NULL DEFAULT 0, verification_token TEXT, verification_expires_at DATETIME, created_at DATETIME DEFAULT CURRENT_TIMESTAMP, updated_at DATETIME DEFAULT CURRENT_TIMESTAMP );
Existerande tabell users uppgraderas. Om befintlig kod använder username kan det behållas som valfritt fält, medan email blir huvudsaklig inloggningsnyckel.

🧪 Migrationsplan

Stegvis rollout med minimal störning

  • 1. Skapa migrering för nya fält i users.
  • 2. Behåll befintlig admin-användare och lägg till email för den.
  • 3. Lägg till env-var ENABLE_SELF_REGISTRATION=true som feature-flag.
  • 4. Implementera register-/verify-/login-endpoints i src/app.py.
  • 5. Lägg till frontend-formulär och bekräftelsesida.
  • 6. Testa i staging med både gamla och nya användare.
  • 7. Sätt ENABLE_SELF_REGISTRATION på true i produktion när valideringen är klar.

🧑‍💻 Frontend: Registreringsflöde

Ett formulär med följande fält:
  • Email (mandatory, valid email)
  • Password (mandatory, min 8 chars)
  • Confirm password (match)
  • Role (optional dropdown med begränsade värden)
Den klientbaserade valideringen bör kontrollera:
  • Giltig e-postadress
  • Lösenord minst 8 tecken
  • Bekräftelselösenord matchar
  • Endast tillåtna roller väljs

⚙️ Backend: Endpoint-design

Följande API-kontrakt föreslås:
POST /api/register Request: { "email": "user@example.com", "password": "SecurePass123", "confirm_password": "SecurePass123", "role": "member" } Response 201: { "message": "Verification email sent. Check your inbox." } GET /api/verify-email?token=... Response 200: { "message": "Email verified. You may now log in." } POST /api/login Request: { "email": "user@example.com", "password": "SecurePass123" } Response 200: { "token": "jwt_token_string" } POST /api/password-reset-request Request: { "email": "user@example.com" } Response 200: { "message": "Password reset email sent." } POST /api/password-reset Request: { "token": "reset_token", "password": "NewSecurePass123", "confirm_password": "NewSecurePass123" } Response 200: { "message": "Password has been reset." }

🔐 Backend: Säkerhetskrav

Grundläggande säkerhetskrav

  • Hasha lösenord med salted hashing via werkzeug.security.generate_password_hash() eller Argo2/bcrypt.
  • Styr e-postverifiering med token och tidsbegränsning.
  • Rate limit för /api/register och /api/login.
  • CSRF-skydd på HTML-formulär och POST-begäranden.
  • Ej exponera detaljerade interna felmeddelanden till klienten.
  • Logga login-försök och verifieringsfel utan att ge ut detaljer som möjliggör kontoenumeration.
Forslag till skydd:
  • Flask-Limiter med 5-10 försök per minut.
  • CSRF-token via Flask-WTF för webbaserade formulär.
  • SameSite=Strict och HttpOnly för cookies.
  • Stäng av detaljerade tracebacks i produktion.

🧩 Rollhantering

Rollfältet får inte vara ett fritt textfält. Endast säkerställda roller ska tas emot, till exempel:
  • member (standard)
  • tester
Admin-rollen måste tilldelas internt eller via ett admin-verktyg. Förhindra admin-registrering med:
  • Frontend: dölja admin från roll-listor.
  • Backend: avvisa role: admin i /api/register.
  • Admin-uppgradering: endast via admin-API med behörighetskontroll.

🧪 Testplan

Nödvändiga tester:
  • Unit: valid register payload skapar användare och token.
  • Unit: ogiltig e-post eller svagt lösenord avvisas.
  • Unit: registrering med redan registrerad e-post ger 409.
  • Integration: verifieringslänk aktiverar kontot.
  • Integration: login efter verifiering returnerar JWT.
  • Integration: login före verifiering avvisas.
  • Integration: lösenordsåterställning skickar email och återställer lösenord.
  • Security: rate limiting blockerar brute force.

💻 Kodförslag

Pseudokod och konkret backend-exempel i Python/Flask. Detta kan läggas in i src/app.py och återanvänder befintlig databaslogik.
from itsdangerous import URLSafeTimedSerializer from werkzeug.security import generate_password_hash, check_password_hash from datetime import datetime, timedelta serializer = URLSafeTimedSerializer(app.config['SECRET_KEY']) ALLOWED_ROLES = ['member', 'tester'] @app.route('/api/register', methods=['POST']) def register(): data = request.get_json(force=True) is_valid, msg = validate_json_payload(data) if not is_valid: return jsonify({'message': msg}), 400 email = data.get('email') password = data.get('password') confirm = data.get('confirm_password') role = data.get('role', 'member') if not email or not password or not confirm: return jsonify({'message': 'Email, password and confirm_password are required'}), 400 if password != confirm: return jsonify({'message': 'Passwords do not match'}), 400 if len(password) < 8: return jsonify({'message': 'Password must be at least 8 characters'}), 400 if role not in ALLOWED_ROLES: return jsonify({'message': 'Role is not allowed'}), 400 email = sanitize_html(email.lower()) conn = get_db() existing = conn.execute('SELECT id FROM users WHERE email = ?', (email,)).fetchone() if existing: return jsonify({'message': 'Email already registered'}), 409 password_hash = generate_password_hash(password, method='pbkdf2:sha256', salt_length=16) token = serializer.dumps(email, salt='email-confirm') expires_at = datetime.utcnow() + timedelta(hours=24) conn.execute( '''INSERT INTO users (email, password, role, is_active, verification_token, verification_expires_at) VALUES (?, ?, ?, 0, ?, ?)''', (email, password_hash, role, token, expires_at) ) conn.commit() conn.close() send_verification_email(email, token) return jsonify({'message': 'Verification email sent. Check your inbox.'}), 201
@app.route('/api/verify-email') def verify_email(): token = request.args.get('token') if not token: return jsonify({'message': 'Token is required'}), 400 try: email = serializer.loads(token, salt='email-confirm', max_age=86400) except Exception: return jsonify({'message': 'Token is invalid or expired'}), 400 conn = get_db() user = conn.execute( 'SELECT id, verification_expires_at FROM users WHERE email = ? AND verification_token = ?', (email, token) ).fetchone() if not user: return jsonify({'message': 'Verification token invalid'}), 400 conn.execute( 'UPDATE users SET is_active = 1, verification_token = NULL, verification_expires_at = NULL WHERE id = ?', (user['id'],) ) conn.commit() conn.close() return jsonify({'message': 'Email verified. You may now log in.'}), 200
@app.route('/api/login', methods=['POST']) def login(): data = request.get_json(force=True) is_valid, msg = validate_json_payload(data) if not is_valid: return jsonify({'message': msg}), 400 email = data.get('email') password = data.get('password') if not email or not password: return jsonify({'message': 'Email and password are required'}), 400 conn = get_db() user = conn.execute('SELECT * FROM users WHERE email = ?', (email.lower(),)).fetchone() conn.close() if not user: return jsonify({'message': 'Invalid credentials'}), 401 if not user['is_active']: return jsonify({'message': 'Account not active. Verify your email.'}), 403 if not check_password_hash(user['password'], password): return jsonify({'message': 'Invalid credentials'}), 401 token = jwt.encode({ 'user': user['email'], 'role': user['role'], 'exp': datetime.utcnow() + timedelta(hours=24) }, app.config['SECRET_KEY'], algorithm='HS256') return jsonify({'token': token}), 200

📤 E-postsändning & verifiering

Använd befintliga e-postinställningar med nya env-vars:
  • EMAIL_HOST
  • EMAIL_PORT
  • EMAIL_USERNAME
  • EMAIL_PASSWORD
  • EMAIL_FROM
  • EMAIL_USE_TLS / EMAIL_USE_SSL
def send_verification_email(email, token): verification_url = f"{app.config['APP_BASE_URL']}/api/verify-email?token={token}" subject = 'Verify your QA Platform account' body = f"Click this link to activate your account:\n\n{verification_url}\n\nThe link expires in 24 hours." send_email( to=email, subject=subject, body=body )

⚙️ Deployment / Konfiguration

Nyckelkonfigurationer som bör sättas via miljövariabler:
  • JWT_SECRET_KEY / SECRET_KEY
  • EMAIL_HOST, EMAIL_PORT, EMAIL_USERNAME, EMAIL_PASSWORD
  • APP_BASE_URL
  • PASSWORD_HASH_METHOD (t.ex. pbkdf2:sha256)
  • PASSWORD_HASH_SALT_LENGTH (t.ex. 16)
  • RATE_LIMIT_LOGIN, RATE_LIMIT_REGISTER
  • ENABLE_SELF_REGISTRATION
Säkerhetsheaders och cookie-inställningar:
  • Strict-Transport-Security
  • X-Frame-Options: DENY
  • X-Content-Type-Options: nosniff
  • Content-Security-Policy
  • SESSION_COOKIE_SECURE = True
  • SESSION_COOKIE_HTTPONLY = True
  • SESSION_COOKIE_SAMESITE = 'Strict'
