TestApiSkills QA Plattform | Datum: 2026-03-25

🎯 Översikt

Detta dokument fokuserar på intervjufrågor inom API och UI automatisering baserat på praktisk erfarenhet från TestApiSkills-projektet. Vi täcker API-testning med dokumentation, kontraktstestning och negativa tester, samt UI-automatisering med Playwright. Varje sektion innehåller relevanta frågor och konkreta svar som visar teknisk kompetens.

📚 API-Testning & Dokumentation

Fråga: Hur implementerade du API-dokumentation i ditt senaste projekt?

Föreslaget Svar: "Jag använde Flask-Swagger (Flasgger) för att skapa interaktiv API-dokumentation. Jag konfigurerade Swagger UI med detaljerade endpoints som inkluderade autentisering (Bearer tokens), request/response-scheman och exempeldata. Detta gjorde det möjligt för utvecklare att testa API:et direkt från dokumentationen på /apidocs/."

Fråga: Vad är API-kontraktstestning och hur använde du det?

Föreslaget Svar: "API-kontraktstestning validerar att API-responses matchar förväntade scheman. Jag använde pytest och jsonschema för att skapa 4 kontraktstester som kontrollerade obligatoriska fält, datatyper och statuskoder. Detta förhindrade brytande ändringar och säkerställde API-stabilitet under utveckling."

Fråga: Hur hanterade du negativa API-tester?

Föreslaget Svar: "För negativa tester skapade jag 17 testfall som täckte ogiltiga inputs, autentiseringsfel och edge cases. Detta inkluderade SQL-injektion, XSS-attacker och boundary values. Jag använde pytest fixtures för att simulera felvillkor och verifiera robust felhantering i applikationen."

Fråga: Vilka verktyg använde du för API-testning?

Föreslaget Svar: "Jag använde pytest som huvudramverk med requests-biblioteket för HTTP-anrop. För schema-validering använde jag jsonschema, och för testdata-generering använde jag Faker. Allt integrerades i en CI/CD-pipeline med HTML-rapporter och täckningsanalys."

Fråga: Hur säkerställde du API-testkvalitet?

Föreslaget Svar: "Jag implementerade olika testtyper: positiva tester för normal funktion, negativa tester för felhantering, och prestandatester för response-tider. Jag använde fixtures för att skapa testdata och säkerställa isolering mellan tester. Resultaten rapporterades med pytest-html och pytest-cov för täckningsanalys."

🖥️ UI-Automatisering

Fråga: Vilket ramverk valde du för UI-automatisering och varför?

Föreslaget Svar: "Jag valde Playwright för dess cross-browser-stöd (Chromium, Firefox, WebKit), auto-waiting-funktioner och moderna API. Det hanterar dynamiska webbapplikationer bättre än Selenium och ger bättre debugging med screenshots och videoinspelningar. Vi implementerade 16 tester som täckte hela användarflöden."

Fråga: Hur hanterade du cross-browser-testning?

Föreslaget Svar: "Jag konfigurerade Playwright att köra tester parallellt över tre webbläsare. I CI/CD-pipeline kördes testerna automatiskt på alla webbläsare för att säkerställa kompatibilitet. Jag hanterade webbläsarspecifika skillnader genom villkorlig logik i testkoden."

Fråga: Beskriv din strategi för UI-testautomatisering.

Föreslaget Svar: "Min strategi inkluderade Page Object Model för att organisera koden, custom fixtures för autentiserade sessioner, och omfattande täckning av användarflöden. Jag skapade tester för login (5 tester), CRUD-operationer (5 tester), sökning (3 tester) och paginering (3 tester). Testerna kördes i både headed (för debugging) och headless-läge."

Fråga: Hur hanterade du dynamiskt innehåll i UI-tester?

Föreslaget Svar: "Playwrights auto-waiting hanterade de flesta dynamiska element automatiskt. För komplexa fall använde jag explicit waits med selectors som var mer robusta än XPath. Jag implementerade också retry-mekanismer för flaky tester och använde screenshots för att debugga timing-problem."

Fråga: Vad var dina största utmaningar med UI-automatisering?

Föreslaget Svar: "De största utmaningarna var att hantera flaky tester på grund av timing och dynamiskt innehåll. Jag löste detta genom att använda mer stabila selectors, implementera retry-logik, och köra tester i olika miljöer. En annan utmaning var att underhålla tester när UI ändrades - här hjälpte Page Object Model att isolera ändringar."

🔧 Verktyg & Tekniker

Fråga: Hur integrerade du API och UI-tester i CI/CD?

Föreslaget Svar: "Jag satte upp en GitHub Actions pipeline som körde alla tester automatiskt vid varje push. API-testerna kördes först (snabbare), följt av UI-tester. Pipeline inkluderade parallell exekvering, HTML-rapporter och kvalitetsgates som blockerade deployment vid testfel."

Fråga: Hur hanterade du testdata för automatisering?

Föreslaget Svar: "Jag implementerade separata databaser för testning med automatisk rensning mellan körningar. För API-tester använde jag Faker för att generera realistiska testdata. För UI-tester skapade jag fixtures som satte upp nödvändiga förvillkor, som autentiserade användare och förifyllda data."

Fråga: Vad är viktigt att tänka på när man skriver automatiserade tester?

Föreslaget Svar: "Viktiga principer inkluderar: hållbarhet (använd stabila selectors), underhållbarhet (Page Object Model), läsbarhet (beskrivande testnamn), och pålitlighet (undvik flaky tester). Tester ska vara snabba att köra, ge tydliga felmeddelanden, och täcka kritiska användarflöden snarare än varje edge case."
